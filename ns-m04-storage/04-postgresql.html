PostgreSQL

Уведомления живут в map[int]notification.Notification. Перезапустил сервер — все данные пропали. Для истории и аудита нужна база данных. Go работает с любой реляционной БД через пакет database/sql — стандартный интерфейс с поддержкой подключаемых драйверов.

Go подключается к PostgreSQL через строку соединения (DSN):

DSN

database/sql: Open и Ping

Стандартный пакет database/sql предоставляет единый интерфейс для работы с реляционными БД. Конкретный драйвер подключается отдельно. У нас уже установлен pgx — он реализует database/sql через вспомогательный пакет:

Go
import (
	"database/sql"
	_ "github.com/jackc/pgx/v5/stdlib"
)

_ перед именем пакета — blank import. Пакет не экспортирует ничего, что ты используешь напрямую. Но его функция init() регистрирует pgx как драйвер для database/sql. Без этого импорта sql.Open("pgx", ...) вернёт ошибку unknown driver "pgx".

sql.Open принимает имя драйвера и DSN:

Go
const dsn = "host=localhost port=5432 user=postgres password=notifier dbname=postgres sslmode=disable"

db, err := sql.Open("pgx", dsn)
if err != nil {
	log.Fatal("sql.Open:", err)
}
defer db.Close()

Важная деталь: sql.Open не подключается к базе данных. Он только разбирает DSN и создаёт объект *sql.DB. Реальное соединение произойдёт при первом запросе. Из этого следствие: sql.Open может вернуть nil ошибку даже если PostgreSQL не запущен.

Чтобы проверить соединение немедленно, есть db.Ping():

Go
if err := db.Ping(); err != nil {
	log.Fatal("db ping failed:", err)
}

db.Ping() отправляет пустой запрос к БД и ждёт ответа. Если PostgreSQL недоступен — вернёт ошибку. Лучше узнать о проблеме при старте, чем получить падение при первом HTTP-запросе.

Добавь в main() перед стартом сервера:

Go
// фрагмент cmd/notifier/main.go
const dsn = "host=localhost port=5432 user=postgres password=notifier dbname=postgres sslmode=disable"

db, err := sql.Open("pgx", dsn)
if err != nil {
	log.Fatal("sql.Open:", err)
}
defer db.Close()

if err := db.Ping(); err != nil {
	log.Fatal("db ping failed:", err)
}

logger.Info("database connected")

Запусти:

Bash
go run ./cmd/notifier/
Output
time=... level=INFO msg="database connected"
time=... level=INFO msg="starting server" addr=:8080

*sql.DB — не одно соединение, а пул. Go открывает и закрывает соединения по мере необходимости. defer db.Close() закрывает весь пул при выходе из main.

Создаём таблицу: db.Exec

db.Exec выполняет SQL без возврата строк. Идеально для DDL (CREATE TABLE) и DML без SELECT:

Go
const createTable = `
CREATE TABLE IF NOT EXISTS notifications (
    id        SERIAL PRIMARY KEY,
    recipient TEXT NOT NULL,
    subject   TEXT NOT NULL,
    body      TEXT,
    channel   TEXT,
    is_urgent BOOLEAN DEFAULT false
)`

if _, err := db.Exec(createTable); err != nil {
	log.Fatal("create table:", err)
}

SERIAL PRIMARY KEY — PostgreSQL автоматически генерирует целочисленный ID при каждой вставке. IF NOT EXISTS — если таблица уже есть, запрос ничего не сломает.

db.Exec возвращает (sql.Result, error). sql.Result содержит LastInsertId и RowsAffected. Для DDL интересует только ошибка — результат отбрасываем через _.

Добавь после db.Ping():

Go
if _, err := db.Exec(createTable); err != nil {
	log.Fatal("create table:", err)
}
logger.Info("table ready")
Output
time=... level=INFO msg="database connected"
time=... level=INFO msg="table ready"
time=... level=INFO msg="starting server" addr=:8080

INSERT: параметры и RETURNING

Есть соблазн подставить значения в SQL через форматирование строки. Добавь временный тест в main():

Go
recipient := "ops@example.com"
query := fmt.Sprintf("INSERT INTO notifications (recipient, subject) VALUES ('%s', 'test')", recipient)
db.Exec(query)

Сейчас работает. Но если recipient содержит одиночную кавычку — o'ps@example.com — SQL сломается. А если кто-то передаст '; DROP TABLE notifications; -- — таблица удалится.

Верни обратно к рабочему варианту — это пример того, как делать не надо. Для подстановки значений используй параметры:

Go
db.Exec(
	"INSERT INTO notifications (recipient, subject) VALUES ($1, $2)",
	"ops@example.com", "Deploy done",
)

$1, $2 — позиционные параметры PostgreSQL. db.Exec передаёт значения отдельно от SQL-текста. БД подставляет их безопасно, не интерпретируя как SQL.

После INSERT нужен ID созданной записи. PostgreSQL возвращает его прямо из INSERT через RETURNING:

Go
var id int
err := db.QueryRow(
	`INSERT INTO notifications (recipient, subject, body, channel, is_urgent)
	 VALUES ($1, $2, $3, $4, $5) RETURNING id`,
	n.Recipient, n.Subject, n.Body, n.Channel, n.IsUrgent,
).Scan(&id)

db.QueryRow выполняет запрос и возвращает одну строку. .Scan(&id) читает значение из этой строки в переменную. RETURNING id превращает INSERT в запрос, возвращающий данные.

SELECT: Query и Scan

Для запросов с несколькими строками — db.Query:

Go
rows, err := db.Query(
	"SELECT id, recipient, subject, body, channel, is_urgent FROM notifications ORDER BY id",
)
if err != nil {
	return nil, fmt.Errorf("query: %w", err)
}
defer rows.Close()

defer rows.Close() — аналог defer f.Close() для файлов. Строки занимают соединение из пула. Без Close соединение не вернётся, пул истощится. Ставится сразу после проверки ошибки, до цикла.

Итерация по результатам:

Go
var result []notification.Notification
for rows.Next() {
	var n notification.Notification
	if err := rows.Scan(&n.ID, &n.Recipient, &n.Subject, &n.Body, &n.Channel, &n.IsUrgent); err != nil {
		return nil, fmt.Errorf("scan: %w", err)
	}
	result = append(result, n)
}

rows.Next() переходит к следующей строке, возвращает true пока строки есть. rows.Scan читает текущую строку в переменные.

Критический момент: порядок аргументов Scan должен совпадать с порядком колонок в SELECT. SELECT идёт id, recipient, subject, body, channel, is_urgent — значит, Scan принимает &n.ID, &n.Recipient, &n.Subject, &n.Body, &n.Channel, &n.IsUrgent. Поменяешь местами два поля одного типа (например, recipient и subject — оба string) — ошибки компиляции не будет, но данные перемешаются молча.

После цикла обязательно проверь rows.Err():

Go
if err := rows.Err(); err != nil {
	return nil, fmt.Errorf("rows: %w", err)
}
return result, nil

rows.Next() возвращает false как при нормальном завершении, так и при ошибке. rows.Err() — единственный способ узнать, что пошло не так в процессе итерации.

Попробуй: напиши insertNotification и getAllNotifications

У тебя есть все кусочки. Напиши обе функции в новом файле cmd/notifier/db.go самостоятельно.

  • rows.Err() после цикла

Go
package main

import (
	"database/sql"
	"errors"
	"fmt"

	"notifier/internal/notification"
)

func insertNotification(db *sql.DB, n notification.Notification) (int, error) {
	const op = "insertNotification"

	var id int
	err := db.QueryRow(
		`INSERT INTO notifications (recipient, subject, body, channel, is_urgent)
		 VALUES ($1, $2, $3, $4, $5) RETURNING id`,
		n.Recipient, n.Subject, n.Body, n.Channel, n.IsUrgent,
	).Scan(&id)
	if err != nil {
		return 0, fmt.Errorf("%s: %w", op, err)
	}
	return id, nil
}

func getAllNotifications(db *sql.DB) ([]notification.Notification, error) {
	const op = "getAllNotifications"

	rows, err := db.Query(
		"SELECT id, recipient, subject, body, channel, is_urgent FROM notifications ORDER BY id",
	)
	if err != nil {
		return nil, fmt.Errorf("%s: %w", op, err)
	}
	defer rows.Close()

	var result []notification.Notification
	for rows.Next() {
		var n notification.Notification
		if err := rows.Scan(&n.ID, &n.Recipient, &n.Subject, &n.Body, &n.Channel, &n.IsUrgent); err != nil {
			return nil, fmt.Errorf("%s: scan: %w", op, err)
		}
		result = append(result, n)
	}
	if err := rows.Err(); err != nil {
		return nil, fmt.Errorf("%s: rows: %w", op, err)
	}
	return result, nil
}

Интегрируем в Server

Текущий Server хранит уведомления в map[int]notification.Notification. Замени поля notifications и nextID на db *sql.DB:

Go
// фрагмент cmd/notifier/main.go
type Server struct {
	db     *sql.DB
	logger *slog.Logger
}

ID теперь генерирует PostgreSQL через SERIAL — nextID больше не нужен.

Обнови createNotification:

Go
// фрагмент cmd/notifier/main.go
func (s *Server) createNotification(w http.ResponseWriter, r *http.Request) {
	const op = "Server.createNotification"

	var n notification.Notification
	if err := json.NewDecoder(r.Body).Decode(&n); err != nil {
		w.Header().Set("Content-Type", "application/json")
		w.WriteHeader(http.StatusBadRequest)
		json.NewEncoder(w).Encode(map[string]string{"error": "invalid JSON"})
		return
	}

	if err := notification.Validate(n); err != nil {
		w.Header().Set("Content-Type", "application/json")
		w.WriteHeader(http.StatusBadRequest)
		json.NewEncoder(w).Encode(map[string]string{"error": err.Error()})
		return
	}

	id, err := insertNotification(s.db, n)
	if err != nil {
		s.logger.Error("insert failed", "op", op, "err", err)
		w.WriteHeader(http.StatusInternalServerError)
		return
	}
	n.ID = id

	w.Header().Set("Content-Type", "application/json")
	w.WriteHeader(http.StatusCreated)
	json.NewEncoder(w).Encode(n)
}

listNotifications теперь просто вызывает getAllNotifications:

// фрагмент cmd/notifier/main.go
func (s *Server) listNotifications(w http.ResponseWriter, r *http.Request) {
	const op = "Server.listNotifications"

	notifications, err := getAllNotifications(s.db)
	if err != nil {
		s.logger.Error("list failed", "op", op, "err", err)

nil слайс маршалится в null. Проверка if notifications == nil гарантирует пустой массив [].

Для getNotification нужна функция поиска по ID. Если запись не найдена, QueryRow.Scan возвращает sql.ErrNoRows:

Go
// фрагмент cmd/notifier/db.go
func getNotificationByID(db *sql.DB, id int) (notification.Notification, error) {
	const op = "getNotificationByID"

	var n notification.Notification
	err := db.QueryRow(
		"SELECT id, recipient, subject, body, channel, is_urgent FROM notifications WHERE id=$1",
		id,
	).Scan(&n.ID, &n.Recipient, &n.Subject, &n.Body, &n.Channel, &n.IsUrgent)
	if errors.Is(err, sql.ErrNoRows) {
		return notification.Notification{}, fmt.Errorf("%s: %w", op, notification.ErrNotFound)
	}
	if err != nil {
		return notification.Notification{}, fmt.Errorf("%s: %w", op, err)
	}
	return n, nil
}

sql.ErrNoRows — sentinel из database/sql для "строка не найдена". Оборачиваем в notification.ErrNotFound чтобы handler мог проверить через errors.Is.

func (s *Server) getNotification(w http.ResponseWriter, r *http.Request) {
	const op = "Server.getNotification"

	idStr := r.PathValue("id")
	id, err := strconv.Atoi(idStr)
	if err != nil {
		w.Header().Set("Content-Type", "application/json")
		w.WriteHeader(http.StatusBadRequest)
		json.NewEncoder(w).Encode(map[string]string{"error": "invalid id"})
		return
	}

	n, err := getNotificationByID(s.db, id)
	if errors.Is(err, notification.ErrNotFound) {
		w.Header().Set("Content-Type", "application/json")
		w.WriteHeader(http.StatusNotFound)
		json.NewEncoder(w).Encode(map[string]string{"error": "notification not found"})
		return
	}
	if err != nil {
		s.logger.Error("get failed", "op", op, "err", err)
		w.WriteHeader(http.StatusInternalServerError)
		return
	}

	w.Header().Set("Content-Type", "application/json")
	json.NewEncoder(w).Encode(n)
}

Частые ошибки:

  • fmt.Sprintf("INSERT ... values='%s'", recipient) вместо $1 — SQL injection, а одиночная кавычка в значении ломает синтаксис SQL
  • rows.Scan(&n.Subject, &n.Recipient, ...) когда SELECT имеет recipient первым — поля тихо меняются местами, ошибки компиляции нет
  • Забыть defer rows.Close() — соединения из пула не возвращаются, следующий db.Query получит timeout
  • db.Ping() без проверки ошибки — сервер стартует без БД, первый POST упадёт

Проверь себя

01sql.Open вернул nil ошибку. Гарантирует ли это что PostgreSQL запущен и доступен?
Ответ

Нет. sql.Open только разбирает DSN и создаёт *sql.DB. Реальное соединение устанавливается при первом запросе к БД. Для проверки доступности используй db.Ping() — он отправляет реальный запрос и вернёт ошибку если PostgreSQL недоступен.

02Почему нельзя подставлять значения в SQL через fmt.Sprintf?
Go
query := fmt.Sprintf("SELECT * FROM notifications WHERE recipient='%s'", recipient)
Ответ

Две проблемы. Во-первых, SQL injection: если recipient = "x' OR '1'='1", запрос вернёт все записи. Во-вторых, синтаксическая ошибка: одиночная кавычка в значении (o'brien@example.com) сломает SQL. Параметры $1, $2 передаются отдельно от SQL-текста — БД принимает их как данные, не интерпретирует как код.

03После цикла for rows.Next() достаточно ли проверить только ошибку db.Query, или нужно ещё rows.Err()?
Ответ

Нужен rows.Err(). db.Query возвращает ошибку только при старте запроса. Если соединение упало в процессе получения строк, rows.Next() вернёт false — как при нормальном завершении — а ошибка окажется в rows.Err(). Без проверки вернёшь неполный список без признака ошибки.

04Можно ли изменить порядок аргументов в rows.Scan если поменять местами два поля одного типа?
Go
// SELECT id, recipient, subject FROM notifications
rows.Scan(&n.ID, &n.Subject, &n.Recipient) // поменяли recipient и subject
Ответ

Компилятор не заметит — оба поля string. rows.Scan читает колонки в порядке SELECT: первая колонка recipient попадёт в &n.Subject, вторая subject — в &n.Recipient. Ошибки не будет, данные перемешаются молча. Порядок аргументов Scan — это контракт с SELECT-запросом, не со структурой.

05Из M4.03: что произойдёт с defer db.Close() если sql.Open вернул ошибку и db == nil?
Ответ

Паника: nil.Close(). defer регистрируется в момент объявления. Если написать defer db.Close() до проверки ошибки sql.Open, и sql.Open вернул ошибку (db == nil), то при выходе из функции вызовется nil.Close() — panic. Порядок: проверь ошибку sql.Open — только потом defer db.Close(). Тот же паттерн что с defer f.Close() после os.Create.

Резюме

Keydatabase/sql и PostgreSQL
  • sql.Open создаёт *sql.DB и разбирает DSN, но не подключается; db.Ping() проверяет реальное соединение
  • _ "github.com/jackc/pgx/v5/stdlib" — blank import регистрирует pgx как драйвер через init()
  • db.Exec для DDL и DML без возврата строк; db.QueryRow для одной строки; db.Query для нескольких
  • $1, $2 параметры вместо форматирования строк — защита от SQL injection
  • RETURNING id в INSERT позволяет получить сгенерированный id одним запросом
  • defer rows.Close() ставится сразу после db.Query — как defer f.Close() после os.Open
  • Порядок аргументов rows.Scan обязан совпадать с порядком колонок в SELECT
  • rows.Err() проверяется после цикла — ошибки итерации попадают туда, не в rows.Next()
  • sql.ErrNoRows — sentinel для "строка не найдена"; оборачивай в domain error для handler-а
