defer — отложенное выполнение

Гарантируем выполнение кода при выходе из функции.

ЗАЧЕМ НУЖЕН DEFER

Часто нужно выполнить "уборку" в конце функции: закрыть файл, освободить ресурс, записать лог. Проблема — функция может завершиться в разных местах:

Go
func process() {
    // Открыли ресурс...

    if condition1 {
        return  // Забыли закрыть!
    }

    if condition2 {
        return  // Снова забыли!
    }

    // Закрыли ресурс... но только если дошли сюда
}

defer решает эту проблему — он гарантирует выполнение кода при выходе из функции.

DEFER НА САМОМ ДЕЛЕ

Что делает defer

Оператор defer откладывает выполнение функции до момента выхода из текущей функции:

Go
package main

import "fmt"

func main() {
    defer fmt.Println("Это выполнится последним")
    fmt.Println("Это выполнится первым")
    fmt.Println("Это выполнится вторым")
}

Вывод:

Output
Это выполнится первым
Это выполнится вторым
Это выполнится последним

Когда выполняется defer

Go
func example() {
    defer cleanup()    // запоминается, НЕ выполняется
    // ...
    // код функции
    // ...
}  // вот здесь выполняется cleanup()
defer выполняется:
При return
При достижении конца функции
При panic

БАЗОВЫЙ СИНТАКСИС

Go
defer выражение

После defer идёт вызов функции:

Go
defer fmt.Println("конец")
defer cleanup()
defer file.Close()

Пример с несколькими return

Go
package main

import "fmt"

func main() {
    result := process(5)
    fmt.Println("Результат:", result)
}

func process(n int) int {
    defer fmt.Println("Выход из process")

    if n < 0 {
        return -1
    }

    if n == 0 {
        return 0
    }

    return n * 2
}

Вывод:

Output
Выход из process
Результат: 10

Независимо от того, какой return выполнится, defer сработает.

НЕСКОЛЬКО DEFER — ПОРЯДОК LIFO

Если в функции несколько defer, они выполняются в обратном порядке — LIFO (Last In, First Out):

Go
package main

import "fmt"

func main() {
    defer fmt.Println("Первый defer")
    defer fmt.Println("Второй defer")
    defer fmt.Println("Третий defer")
    fmt.Println("Основной код")
}

Вывод:

Output
Основной код
Третий defer
Второй defer
Первый defer
Стек defer (LIFO)
defer A → [A]
defer B → [A, B]
defer C → [A, B, C]
Выполнение: C → B → A

АРГУМЕНТЫ ВЫЧИСЛЯЮТСЯ СРАЗУ

Важно: аргументы defer-функции вычисляются в момент defer, а не в момент выполнения:

Go
package main

import "fmt"

func main() {
    x := 10
    defer fmt.Println("Значение x:", x)  // x вычисляется здесь!
    x = 20
    fmt.Println("Текущее x:", x)
}

Вывод:

Output
Текущее x: 20
Значение x: 10

Значение x (10) было "захвачено" в момент defer, хотя функция выполнилась позже.

ПРАКТИЧЕСКОЕ ПРИМЕНЕНИЕ

Логирование входа/выхода

Go
package main

import "fmt"

func main() {
    doWork()
}

func doWork() {
    fmt.Println("Начало doWork")
    defer fmt.Println("Конец doWork")

    fmt.Println("Выполняем работу...")
}

Вывод:

Output
Начало doWork
Выполняем работу...
Конец doWork

Закрытие ресурсов (концепция)

Типичный паттерн с файлами (подробнее в модуле о файлах):

Go
func processFile(filename string) {
    // Открываем файл
    // ...

    defer closeFile()  // Гарантированно закроем

    // Работаем с файлом
    // Даже если здесь ошибка — файл закроется
}

Отмена операции при ошибке

Go

import "fmt"

func main() {
    transaction()
}

func transaction() {
    fmt.Println("Начинаем транзакцию")
    defer fmt.Println("Завершаем транзакцию")

    fmt.Println("Шаг 1: готово")
    fmt.Println("Шаг 2: готово")
    fmt.Println("Шаг 3: готово")
}

Вывод:

Output
Начинаем транзакцию
Шаг 1: готово
Шаг 2: готово
Шаг 3: готово
Завершаем транзакцию

ЛОВУШКИ DEFER

Ловушка 1: defer в цикле

Go
package main

import "fmt"

func main() {
    for i := 0; i < 3; i++ {
        defer fmt.Println(i)
    }
    fmt.Println("После цикла")
}

Вывод:

Output
После цикла
2
1
0

Все defer накапливаются и выполняются при выходе из функции, не из цикла!

Ловушка 2: изменение переменной после defer

Go
package main

import "fmt"

func main() {
    x := 1
    defer fmt.Println(x)  // Захватывает x = 1
    x = 2
}

Вывод: 1

Аргумент вычислился в момент defer, когда x был равен 1.

Ловушка 3: забыть скобки

Go
defer fmt.Println  // Неправильно! Нужно: defer fmt.Println()

После defer должен быть вызов функции.

DEFER С АНОНИМНОЙ ФУНКЦИЕЙ

Чтобы defer использовал актуальное значение переменной, можно использовать анонимную функцию:

Go
package main

import "fmt"

func main() {
    x := 1
    defer func() {
        fmt.Println(x)  // x вычисляется при выполнении defer
    }()
    x = 2
}

Вывод: 2

Анонимная функция "захватывает" переменную x, а не её значение.

ПРОВЕРЬ СЕБЯ

Вопросы

01Когда выполняется код после defer?
Показать ответ

Код после defer выполняется при выходе из функции: при return, при достижении конца функции, или при panic.

02В каком порядке выполняются несколько defer?
Показать ответ

В обратном порядке — LIFO (Last In, First Out). Последний defer выполнится первым.

03Когда вычисляются аргументы defer-функции?
Показать ответ

Аргументы вычисляются в момент выполнения defer, а не в момент вызова отложенной функции. Это важно помнить при работе с переменными.

04Выполнится ли defer, если функция завершается через return?
Показать ответ

Да. defer выполняется при любом выходе из функции, включая return.

РЕЗЮМЕ

KeyКлючевые идеи
  • defer откладывает выполнение до выхода из функции
  • Несколько defer выполняются в обратном порядке (LIFO)
  • Аргументы вычисляются в момент defer, не при выполнении
  • defer выполняется при любом способе выхода из функции

Главная ловушка: Ожидать актуальное значение переменной в defer — аргументы "замораживаются" в момент defer.

Правило: Используйте defer для гарантированной очистки ресурсов — он выполнится независимо от пути выхода из функции.
